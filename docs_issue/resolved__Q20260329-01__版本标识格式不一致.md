# 问题标题
版本号命名格式在 API、Schema 与示例之间不一致

# 问题摘要
版本标识格式不一致

# 问题描述
当前设计文档对 version 标识的格式没有收敛为单一规范。`POST /contents/{content_id}/rollback` 的请求示例使用 `"version": "v3"`，而 `content.<version>.json` 的示例中 `version` 字段又使用 `"1"`，同时文件命名采用 `content.<version>.json` 的占位表达。当前行为会导致实现层无法确定 version 的真实规范到底是纯数字、带 `v` 前缀的字符串，还是任意 label 风格字符串。

# 严重程度
Medium

# 影响对象
`docs_design/05-api-design.md`、`docs_design/09-content-json-schema.md`、版本创建/查询/回滚接口实现

# 问题原因
在引入 `content_versions` 之后，版本元数据、文件命名和 API 参数被分散描述，但没有单独定义“版本号”和“展示标签”的职责边界，导致示例各自演化，最终留下不一致格式。

# 核心证据路径
docs_design/05-api-design.md

# 待确认差异
无

- 日志/报错：N/A

- 复现步骤：
  1. 阅读 `POST /contents/{content_id}/rollback` 请求示例，发现使用 `"version": "v3"`。
  2. 阅读 `content.<version>.json` 的 version 示例，发现使用 `"version": "1"`。
  3. 无法从文档判断 `content_versions.version`、对象存储命名和 API 参数是否必须完全一致。

# 影响
实现方可能在数据库、对象存储命名、路由参数解析和前端调用中分别采用不同格式，造成版本查询失败、回滚找不到目标版本，或后续需要做兼容迁移。

# 建议解决方案
明确拆分两个概念：

1. `version`：机器可比对的规范化版本号，例如仅允许数字字符串 `1`、`2`、`3`。
2. `label`：面向展示的可选标签，例如 `v1`、`2026Q1`。

随后统一以下文档：

1. API 请求参数中的 `version`
2. `content_versions.version`
3. `content.<version>.json` 命名规则
4. 示例 JSON 中的 `version` 字段

# 验收标准
1. `docs_design` 中所有 `version` 的机器字段示例都使用同一种格式。
2. 文档明确区分 `version` 与 `label` 的职责，不再混用。
3. 版本创建、查询、回滚三个 API 的参数语义可以从文档直接推导，无需额外猜测。

# 评论
content.*.json 文件名的version可以是 0补全+数字=6位数字的形式
数据中或者请求中的version都是纯整数
任何一个地方都不存在`v+版本号`的写法
label是用户编写的与版本管理没有直接关系
