# 问题标题
从 latest 生成 version 的流程缺少状态转换和字段重写步骤

# 问题摘要
版本生成缺转换步骤

# 问题描述
版本设计文档当前写法是“读取当前 `content.latest.json`，复制为指定版本号的快照，然后写入 `content.<version>.json`”。但 `content.latest.json` 与 `content.<version>.json` 的 JSON Schema 并不相同，后者要求 `state=version` 且 `version` 必填。若实现方按字面执行“复制”，将得到一个仍然带有 `state=latest` 或 `version=null` 的非法版本文件。

# 严重程度
High

# 影响对象
`docs_design/03-versioning-and-publishing.md`、`docs_design/09-content-json-schema.md`、版本创建实现

# 问题原因
版本流程文档强调了“从 latest 衍生 version”的来源关系，但没有把“结构转换步骤”明确写出来，导致流程描述与 JSON Schema 约束之间存在落差。

  3. 文档没有显式说明需要将 `state` 从 `latest` 改为 `version`，并写入具体 `version` 值。

# 影响
该问题会直接影响版本文件合法性。若实现不补这一步，生成的 `content.<version>.json` 无法满足 schema，进而影响回滚、分发、校验和审计。

# 建议解决方案
在版本创建流程中显式增加结构转换步骤：

1. 读取 `content.latest.json`
2. 深拷贝节点数组
3. 将顶层 `state` 改为 `version`
4. 写入规范化 `version`
5. 可选更新 `label`
6. 写入 `content.<version>.json`
7. 落库 `content_versions`

同时在 API 文档和 JSON Schema 文档中补一条说明：`version` 文件是由 latest 快照“派生并重写头部字段”得到，而不是字节级复制。

# 验收标准
1. 版本创建流程文档显式包含 `state` 和 `version` 的重写步骤。
2. API、版本设计、JSON Schema 三份文档对 version 文件生成方式表述一致。
3. 读者按照文档实现时，不会生成 `state=latest` 的 version 文件。
